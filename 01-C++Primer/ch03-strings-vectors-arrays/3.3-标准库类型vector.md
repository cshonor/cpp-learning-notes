# 3.3 标准库类型 vector

## 本节讲什么

介绍 C++ 标准库中最常用的**容器**类型 **`std::vector`**——存放**同类型元素**的**动态数组**（可变长序列）。讲解 **`()` 与 `{}` 两种初始化**（新手高频坑）、**`push_back`** 动态增容，以及下标访问规则（**不能用下标越界添加元素**）。

## 名字别搞混：vector ≠ 宿主

| 英文 | 常见中文 | 在 C++ / HFT 里指什么 |
|------|----------|------------------------|
| **vector** | **向量、容器** | `std::vector<T>`：标准库动态数组，本章主题 |
| **host** | **宿主** | 网络/系统语境：本机、宿主机（如 host byte order、宿主机 NIC） |

`vector` 来自数学「向量」的命名，在 C++ 里语义是**一维元素序列容器**，和网络里的「宿主机器」毫无关系。

## 要点

- `#include <vector>`；`std::vector<T>` 存放类型 `T` 的对象序列（**元素类型必须相同**）
- **动态数组**：长度不固定，元素在内存中**连续存放**（对缓存友好；HFT 批量 tick 常用）
- **`push_back`**：在末尾追加元素，是运行时**增容、添加新元素**的正确方式
- **下标从 0 开始**：合法下标 `0 … size()-1`；**`v[v.size()]` 永远越界**（见下文）
- **不能用下标添加**：`v[10] = 42;` 在 `size() <= 10` 时未定义行为——容器不会自动扩容，只是往非法地址写
- 可用 `size()`、`empty()` 查询状态；增删改查 API 见第 9 章顺序容器

## 小括号 `()` vs 花括号 `{}`：两种初始化（C++11）

这是 **`vector` 最容易混的两套逻辑**，刚好对应你看到的 **小括号** 和 **花括号**：

| | **小括号 `()`** | **花括号 `{}`** |
|---|----------------|-----------------|
| 叫法 | **默认值 / 个数初始化** | **列表初始化**（C++11 **统一初始化**语法的一部分） |
| 语义 | 「给我 **n 个**元素，每个按规则填初值」 | 「花括号里 **有几个数，容器就有几个元素**」 |
| 典型 | `v(10)` → 10 个 `0` | `v{1,2,3}` → 3 个元素，就是 1、2、3 |
| 自定义类 | 每个元素调 **默认构造函数** | 用列表里的值 **逐个构造**元素，**不会**多生成默认对象 |

混用是新手最常踩的坑；HFT 里 **`size()` 错一位**就可能把行情缓冲长度搞错，后续循环读写全偏。

### 圆括号：个数 + 默认值初始化

**`vector<int> v(10)`** 的意思就是：**给我 10 个 `int`，每个都用 `int` 的默认值 `0` 填充**。

语法是「**我要 n 个元素，每个怎么初始化**」：

| 写法 | 含义 | `size()` |
|------|------|----------|
| `vector<int> v(10);` | 分配 **10 个 `int`**，每个 **值初始化** → `0` | 10 |
| `vector<int> v(10, 5);` | **10 个 `int`**，每个都是 **`5`** | 10 |
| `vector<MarketData> v(100);` | **100 个对象**，每个调用 **`MarketData` 默认构造** | 100 |

```cpp
std::vector<int> v(10);      // 10 个 0
std::vector<int> v2(10, 5);  // 10 个 5

struct MarketData { double bid{0}, ask{0}; };
std::vector<MarketData> book(100);  // 100 个默认构造的 tick 槽位
```

### 花括号：列表初始化（挨个塞初值）

**`vector<int> v{1, 2, 3}`** 是 C++11 的 **列表初始化**：逻辑是 **把花括号里的元素挨个当作 `vector` 的初值**——这里 `v` **只有 3 个元素**，值就是 `1、2、3`，**不会**再额外生成默认值填充空位。

语法是「**容器里就只有这些元素**」——**不会**因为写了一个数字就自动扩成 n 个默认值：

| 写法 | 含义 | `size()` |
|------|------|----------|
| `vector<int> v{1, 2, 3};` | 恰好 **3** 个元素：`1, 2, 3` | 3 |
| `vector<int> v{10, 5};` | 恰好 **2** 个元素：`10` 和 `5` | **2**（不是 10 个 5！） |
| `vector<int> v{10};` | 恰好 **1** 个元素：`10` | 1 |

```cpp
std::vector<int> a(10, 5);   // size=10，全是 5
std::vector<int> b{10, 5};   // size=2，元素是 10 和 5  ← 最容易写反
```

### 对照记忆

```
v(10)      → 10 个元素，默认填 0（或默认构造）
v(10, 5)   → 10 个元素，全是 5
v{1,2,3}   → 3 个元素，就是 1、2、3
v{10, 5}   → 2 个元素，就是 10 和 5   （不是 v(10,5)！）
```

### HFT：长度错了 = 静默读错行情

- 预分配 **100 档深度**：应写 `vector<Level>(100)` 或 `vector<Level>(100, default_level)`，**不要**写成 `vector<Level>{100}`（那只有 **1** 个元素）
- 从配置文件读入 **N 个价格**：用 `{p1, p2, ...}` 列表或 `push_back`；别用 `v(N)` 误当成「填 N 这个数」
- 写完后习惯 **`assert(v.size() == expected)`** 或单元测试锁死 `size()`，避免热路径带着错误长度跑整天

## 下标从 0 开始：为什么 `v[v.size()]` 越界

`vector` 和 C 数组一样，**第一个元素下标是 `0`**，最后一个合法下标是 **`size() - 1`**。

以 **`vector<int> v{1, 2, 3}`** 为例：

| | 下标 `0` | 下标 `1` | 下标 `2` | 下标 `3` |
|---|----------|----------|----------|----------|
| 元素 | `1` | `2` | `3` | ❌ **不存在** |
| | | | `size()-1` | **`size()` 返回 3 → 越界** |

- **`v.size()`** 返回的是 **元素个数**（这里是 `3`），不是「最大合法下标」
- 合法访问：`v[0]`、`v[1]`、`v[2]`；循环常写 **`i < v.size()`** 或 **`i != v.size()`**
- **`v[v.size()]`** 刚好踩在「第 4 个位置」——容器里根本没有这格，读是 UB，写是 **OOB 写**（笔记里标红的坑）
- **不能用 `v[v.size()] = x` 来「追加」**：`vector` 不会因此扩容，只会往非法地址写

```cpp
std::vector<int> v{1, 2, 3};
// v[3] = 4;           // ❌ 越界：合法下标只有 0,1,2
// v[v.size()] = 4;    // ❌ 同上，size()==3
v.push_back(4);        // ✅ 正确追加
```

## 尖括号里放什么类型？

`vector<T>` 的 **`T` 不限于基本类型**——只要满足容器对元素类型的要求（默认可**拷贝构造、拷贝赋值、可析构**；C++11 起移动语义参与增容），**任何类型**都能做元素：

| `T` 的例子 | HFT 场景 |
|------------|----------|
| `int`、`double` | 价格、数量、时间戳 |
| 自定义类 `MarketData` | 一条 tick / 快照 |
| `Tick*`、`std::unique_ptr<Order>` | 指针或智能指针（注意所有权语义） |
| `std::vector<int>` | **`vector<vector<int>>`**：二维结构，如按档位存的订单簿深度 |

```cpp
struct MarketData {
    double bid{0}, ask{0};
    std::int64_t ts{0};
};

std::vector<MarketData> book;           // 自定义类
std::vector<MarketData*> pending;       // 指针（不拥有对象，需别处管理生命周期）
std::vector<std::vector<int>> levels;  // 嵌套容器：levels[price_level][qty...]
```

### 自定义类型：拷贝 / 移动与扩容

`vector` **扩容**（`push_back` 触发 `reallocate`）时，通常要把已有元素**搬到新内存**——依赖元素的**拷贝或移动**：

- 类型**不可拷贝且不可移动** → 无法放入 `vector`（编译报错）
- 只有拷贝、移动很慢（大 `string`、持锁对象）→ 扩容代价高，热路径抖动
- **优先 `emplace_back`**：在容器内**原地构造**，少一次临时对象拷贝

#### C++ 如何实现「可复制」

C++ 靠 **拷贝构造函数** 和 **拷贝赋值运算符** 表达「可复制」语义：

| 机制 | 作用 |
|------|------|
| **拷贝构造** `T(const T&)` | `vector` 扩容搬元素、按值传参、`push_back(x)` 时可能调用 |
| **拷贝赋值** `operator=(const T&)` | 已有元素被覆盖时调用 |

**你没写时**，编译器会 **`= default` 隐式生成**——对成员做**逐成员拷贝**，等价于**浅拷贝**：

- 成员全是 `int`、`double` 等标量 → 浅拷贝 = 深拷贝，**可直接进 `vector`**
- 类里含 **裸指针 / 动态内存** → 浅拷贝两个对象共享同一块堆内存 → **双析构、悬垂指针**；`vector` 扩容一拷贝就炸

**有指针或动态资源的类**必须手写 **深拷贝**（拷贝构造 + 拷贝赋值 + 析构，即三/五法则，见 [13.1 拷贝控制成员](../ch13-copy-control/13.1-拷贝控制成员.md)）。

```cpp
// POD 风格 tick：编译器生成的拷贝就够
struct MarketData {
    double bid{0}, ask{0};
    std::int64_t ts{0};
    MarketData() = default;
    MarketData(const MarketData&) = default;            // 显式允许拷贝
    MarketData& operator=(const MarketData&) = default;
};

std::vector<MarketData> v;
v.push_back({100.5, 100.6});   // 扩容时逐成员拷贝，安全且便宜
```

#### 和 Rust `Copy` / `Clone` 的对应关系

| Rust | C++ 大致对应 |
|------|----------------|
| **`Copy`**（位拷贝、无所有权转移） |  trivially copyable / POD 标量 struct；编译器隐式拷贝 |
| **`Clone`**（显式 `.clone()`） | 用户定义的拷贝构造 / 拷贝赋值（可能深拷贝） |
| **移动 / 所有权转移** | 移动构造 `T(T&&)` + 移动赋值；`= delete` 拷贝后只留移动 |

Rust 用 **trait 显式标记**能力；C++ **没有**统一 trait，靠你是否声明（或 `= delete`）拷贝/移动函数来**手动控制**类型行为——思路相通，C++ 更啰嗦。

#### HFT：禁用拷贝、只允许移动

热路径大对象或持资源类型，常 **删掉拷贝、保留移动**，让 `vector` 扩容时走 **移动** 而非深拷贝（C++11 起 `vector` 增容优先移动）：

```cpp
struct HeavyBook {
    std::vector<double> levels;
    HeavyBook() = default;
    HeavyBook(HeavyBook&&) = default;
    HeavyBook& operator=(HeavyBook&&) = default;
    HeavyBook(const HeavyBook&) = delete;              // 禁止拷贝
    HeavyBook& operator=(const HeavyBook&) = delete;
};

std::vector<HeavyBook> books;
books.push_back(HeavyBook{});  // 移动进 vector；扩容时移动元素，无深拷贝尖刺
```

更极端：**元素本身不可移动**时，用 `vector<unique_ptr<T>>`，扩容只搬指针（见上文 POD 小节）。

### HFT：为什么常选 POD 作元素

**POD**（Plain Old Data，Plain Old **Data**）：语义上接近 C 结构体——基本类型或**只含基本类型 / POD 成员**的 `struct`，布局简单、无用户自定义拷贝逻辑。

| POD / 轻量 struct | 重对象（大 string、mutex 成员等） |
|-------------------|-----------------------------------|
| 扩容时 **memcpy 级**拷贝，成本低 | 扩容触发深拷贝、分配，延迟尖刺 |
| 缓存行友好，预取命中高 | 元素大，cache miss 多 |
| HFT tick、档位、固定字段快照常用 | 冷路径配置、日志可接受 |

工程上常见组合：

1. **热路径**：`vector<Tick>`，`Tick` 全是标量 / 固定数组（POD 风格）
2. **容量已知**：构造前 **`reserve(n)`**，尽量只扩一次
3. **大对象或非拷贝**：`vector<unique_ptr<T>>` 或 **`vector<T*>`**（指针数组，扩容只搬指针）

## HFT：越界写 = 静默灾难

行情处理热路径里，`vector` 误用下标「添加」元素会导致**缓冲区外写（OOB write）**：

- **不一定会立刻 crash**：可能踩坏相邻对象、计数器、指针，表现为偶发错价、漏单
- **一旦崩溃往往在最坏时刻**：GC 压力、锁竞争、网络突发时，直接**错过交易窗口**
- **工程习惯**：
  - 新元素一律 **`push_back` / `emplace_back`**
  - 下标前断言或边界检查：`i < v.size()`
  - 容量可预估时用 **`reserve(n)`** 减少重分配（见 [3.2.3 拼接与内存优化](./3.2.3-拼接与内存优化.md)）

```cpp
// 行情缓冲：错误 vs 正确
std::vector<Tick> ticks;       // 假设已有 3 条，下标 0..2
// ticks[ticks.size()] = tick; // ❌ size()==3，下标 3 越界，OOB 写
ticks.push_back(tick);         // ✅
// ticks.emplace_back(/* ... */); // ✅ 原地构造
```

## 示例

```cpp
#include <iostream>
#include <vector>

struct MarketData {
    double bid{0}, ask{0};
};

int main() {
    std::vector<int> v(10);          // 10 个 0（圆括号：个数 + 默认值）
    std::vector<int> v_paren(10, 5); // 10 个 5
    std::vector<int> v_list{1, 2, 3}; // 花括号：3 个元素 1,2,3
    // vector<int> v_trap{10, 5};   // 坑：size=2，不是 10 个 5

    for (int i = 1; i <= 5; ++i)
        v.push_back(i);              // 在已有 10 个 0 后再追加

    std::vector<MarketData> ticks;   // 自定义类
    ticks.push_back({100.5, 100.6});

    std::vector<std::vector<int>> book(5);  // 5 档，每档一个 vector
    book[0].push_back(100);

    std::vector<int> v2{1, 2, 3};    // 列表初始化：size=3

    for (std::size_t i = 0; i != v.size(); ++i)
        std::cout << v[i] << " ";    // 下标访问已有元素
    std::cout << std::endl;

    // 错误示例（不要这样做）：
    // v[10] = 100;  // 越界！不能用下标添加新元素

    return 0;
}
```
